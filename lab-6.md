# Лабораторная работа 6. Обработка HTTP-запросов

Цель лабораторной работы — познакомиться с сущностью `Promise` и принципами выполнения асинхронного кода в JS, научиться создавать HTTP-запросы и корректно обрабатывать ответы.

В качестве практикума предлагается добавить "живых" данных на уже имеющийся прототип. В качестве источника данных используйте сервис с mock-данными, например, [JSONPlaceholder](https://jsonplaceholder.typicode.com/) (работает только через VPN). При желании можете найти другой сервис или сделать себе небольшую API [в сервисе от автора JSONPlaceholder](https://my-json-server.typicode.com/).

Перед началом работы необходимо выбрать ту часть проекта, в которую в дальнейшем будут загружаться данные. Это может быть профиль пользователя, прошедшего авторизацию, комментарии под постами, список пользователей в одной из уже созданных таблиц и т.п.

В качестве данных, которые предоставляет сервис из примера, предлагается несколько видов ресурсов на выбор:

- Посты для новостной ленты (`/posts`) и комментарии под ними (`/comments`);
- Альбомы (`/albums`) и фотографии в них (`/photos`);
- Список дел (`/todos`);
- Пользователи (`/users`).

Требования к реализации:

1. Весь код должен быть реализован в отдельном скрипте и подключен на странице с сетевым взаимодействием.

2. Данные должны подтягиваться либо сразу после загрузки страницы, либо при нажатии на определённую кнопку. Запрос должен выполняться с использованием [Fetch API](https://developer.mozilla.org/ru/docs/Web/API/Fetch_API).

3. С момента начала запроса и до получения ответа вместо контента должна отображаться анимация (т.н. прелоадер, либо обычная GIF-анимация, либо CSS-анимация). После получения контента и его отображения прелоадер должен скрываться.

4. При каждом получении данных должна применяться разная фильтрация, можно сделать её случайной (например, каждый раз запрашивать посты случайной категории). При этом фильтрация должна быть на стороне сервера, т.е. не нужно скачивать всё и фильтровать это на клиенте.

5. Все ошибки должны корректно обрабатываться. В качестве ошибок может быть недоступность сети, недоступность API, недоступность конкретного запрашиваемого ресурса. В случае ошибки на странице вместо контента должно отображаться сообщение об ошибке, в идеале для каждого типа своё.

6. Как и в прошлой ЛР, шаблон для добавляемых элементов должен быть вынесен на страницу, а не вшит в скрипт.
